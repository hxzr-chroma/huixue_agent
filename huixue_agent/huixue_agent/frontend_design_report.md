# 前端设计报告

## 1. 设计目标

本系统前端面向“学习规划、学习执行、学习优化”的完整流程进行设计，目标是在保证功能完整的基础上，提供清晰、易理解、适合课程展示的交互体验。前端采用单应用、多页面导航的组织方式，以降低用户首次使用时的理解成本，并使系统使用路径与项目核心逻辑保持一致。

前端设计遵循以下原则：

- 功能流程清晰：页面顺序与业务流程一致
- 信息分层明确：当前计划、进度反馈、学习检测、动态调整相互独立
- 交互负担较低：用户每一步只聚焦当前任务
- 视觉风格简洁：避免复杂装饰，突出产品原型感和可展示性

## 2. 页面组织结构

系统前端采用侧边栏导航加主内容区布局，主要包括以下页面：

1. 首页总览
2. 学习计划生成
3. 当前学习计划
4. 学习进度反馈
5. 学习检测
6. 动态调整

其中，首页用于介绍系统能力和当前状态；其余页面围绕学习辅助主流程逐步展开。

## 3. 页面功能设计

### 3.1 首页总览

主要功能：

- 展示系统名称、核心能力和使用说明
- 显示当前是否已经存在学习计划
- 给出推荐操作路径
- 在有计划时展示关键统计信息，例如当前计划状态、最近检测情况等

设计目的：

- 让用户进入系统后快速理解产品定位
- 作为整个系统的统一入口，增强展示感

### 3.2 学习计划生成页

主要功能：

- 输入学习目标自然语言描述
- 调用学习需求解析和计划生成能力
- 展示结构化目标和生成后的学习计划
- 保存当前学习计划

交互逻辑：

- 若当前无计划，优先引导用户完成该页面操作
- 若已有计划，可在页面中重新生成计划，但默认不强制用户重复输入

### 3.3 当前学习计划页

主要功能：

- 展示当前计划摘要
- 展示阶段安排
- 展示每日任务
- 展示里程碑信息

设计目的：

- 让用户在不进入编辑流程的情况下查看已有学习安排
- 提升“计划可见、执行可追踪”的体验

### 3.4 学习进度反馈页

主要功能：

- 填写学习完成率
- 填写已完成任务、未完成任务
- 填写偏差原因和学习备注
- 生成学习反馈信息

交互逻辑：

- 用户提交进度后，系统即时反馈当前学习状态
- 如学习偏离计划，可引导进入动态调整页面

### 3.5 学习检测页

主要功能：

- 基于最近一次学习内容生成检测题
- 展示基础理解、概念理解、应用思考三类问题
- 提交检测得分、答题简述和自我总结
- 保存检测结果

设计目的：

- 强化系统不是单纯“出计划”，而是“能评估学习效果”
- 为后续优化策略提供依据

### 3.6 动态调整页

主要功能：

- 基于学习进度和检测结果进行综合分析
- 输出学习策略调整建议
- 展示调整后的每日任务安排

设计目的：

- 体现系统的闭环优化能力
- 强化项目中的智能决策特征

## 4. 页面跳转设计

页面跳转采用侧边栏导航菜单实现，用户可直接切换到目标页面。为保证逻辑合理性，各页面在内容上设置前置条件提示：

- 无学习计划时，进度反馈、检测、动态调整页面提示先生成计划
- 无进度记录时，检测页面提示先提交学习进度
- 无检测记录时，动态调整页面仍可使用，但会说明评估依据较少

这种设计既允许自由跳转，也保留业务流程约束，适合演示和后续扩展。

## 5. 页面原型说明

### 5.1 布局结构

- 侧边栏：系统导航、当前状态、使用提示
- 顶部区域：页面标题、副标题
- 主体内容区：表单、卡片、结果展示区

### 5.2 视觉风格

- 主色调采用深蓝与浅灰组合，突出理性与学习产品气质
- 通过卡片容器、圆角、间距和层次增强页面整洁度
- 尽量控制单页信息量，提升可读性

## 6. 前端与业务逻辑对应关系

前端页面与后端服务层接口关系如下：

- 学习计划生成页 -> `StudyPlannerService.create_plan()`
- 当前学习计划页 -> `StudyPlannerService.get_current_plan()`
- 学习进度反馈页 -> `StudyPlannerService.record_progress()`
- 学习检测页 -> `StudyPlannerService.generate_evaluation()` 与 `StudyPlannerService.save_evaluation_result()`
- 动态调整页 -> `StudyPlannerService.adjust_plan()`

这种映射方式保证了页面职责清晰，且前端不直接耦合智能体内部实现。

## 7. 实现说明

前端采用 `Streamlit` 实现，原因如下：

- 开发效率高，适合课程项目快速原型实现
- 与 Python 服务层结合紧密，便于直接调用业务逻辑
- 支持表单、状态保存、布局组件和快速展示

虽然当前未采用前后端分离框架，但通过页面分区、侧边栏导航和统一样式设计，已经具备基本的产品原型形态。

## 8. 后续优化方向

- 引入用户身份识别，实现多用户计划隔离
- 增加历史记录页面，支持查看历次调整结果
- 增加图表化展示，如完成率趋势、检测得分趋势
- 引入更细粒度的页面组件设计，提升整体视觉表现

## 9. RAG 与 LangGraph（后端增强，与前端对应）

- **RAG**：本地知识库目录 `data/knowledge/`，检索结果在服务端注入「计划生成 / 学习检测 / 动态调整」的提示词；前端侧边栏可展示当前知识库片段数量，便于演示时说明数据来源。
- **LangGraph**：学习计划创建与动态调整在服务端通过有向工作流编排（解析 → 检索 → 生成计划；检索 → 优化），与前端页面流程一致，用户仍通过原有菜单完成操作，无需额外跳转。
- 技术说明详见仓库内 `RAG_LANGGRAPH.md`。
